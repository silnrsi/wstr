Transliteration is one of two processes by which text is converted from one script into another. The other process is transcription. The primary aim of transliteration is to provide an alternate means of reading text using a different script. Transliteration is intended to broadly preserve the sounds of the original script, but the focus is not on providing an accurate phonemic representation as it is in the case of transcription. There is often more than one system for transliterating between two given scripts. Transliteration can be between any two scripts, though the term is often used to imply transliteration into the Latin script (which is technically Romanization).

In practice, there is some overlap between the concepts of transcription and transliteration, and some systems in use are neither fully one nor fully the other. However, it is still useful to understand the differences between the two. Broadly speaking, transliteration can be defined as a graphemic conversion from one script to another, and transcription can be defined as a phonemic conversion from one writing system to another. Transcription does not necessarily involve a change in script (for example, in the case of an IPA transcription of a German text, the Latin script is used for the original and the transcribed text). See [Transliteration vs. Transcription](/articlelib/t/transliteration-vs-transcription/) for a more detailed discussion of the differences and similarities between these two processes.

Strictly speaking, a transliteration system ideally uses a one-to-one character mapping so that the orthographic rules of the original script are kept intact. So in the case of a Thai Romanization system, each Thai grapheme will be assigned one Latin counterpart. Generally the Latin transliteration will give some indication as to the pronunciation of the Thai word, but the focus is not on an exact phonemic representation. This one-to-one transliteration is called ‘reversible’ or ‘lossless’ because it allows a transliterated word to be correctly transliterated back into the original script. However, in practice, the term transliteration is used for a wide variety of mappings from one script to another, not only for one-to-one mappings. These ‘irreversible’ systems (sometimes called ‘lossy’ or ‘ambiguous’) do not allow an accurate re-transliteration back into the original script because one Latin character may correspond fully or partially to multiple characters in the source script or vice-versa. For example, the Greek character PSI (:usv[03C8]{char}) can be transliterated in Latin as _ps_, but _ps_ could also result from the sequence PI, SIGMA (:usv[03C0]{char}:usv[03C3]{char}) since PI (:usv[03C0]{char}) can be transliterated as _p_ and SIGMA (:usv[03C3]{char}) can be transliterated as _s_. Transliteration rules have to be carefully designed to avoid lossy transliterations. One way around this is to use an apostrophe (or some other mark) to indicate where an ambiguous transliterated sequence does not result from a single character in the source script. Using this system, in the previous example PSI (:usv[03C8]{char}) would be transliterated as _ps_ but the sequence PI, SIGMA (:usv[03C0]{char}:usv[03C3]{char}) would be transliterated as _p’s_.

The creation of a transliteration standard is often more a case of balancing a predictable approximation of pronunciation against a one-to-one character mapping, than of strict adherence to a fixed set of rules. Some transliteration systems will prioritize reversibility, and will assign one target character to one source character even if the pronunciations are ill-matched, and others will prioritize a predictable pronounceability even if this means the system is not reversible.
